## 1. الحكم التنفيذي (الخلاصة أولاً)

**المشروع قابل للبناء، والقرار هو: ابدأ — بشروط.**

المعمارية ناضجة بدرجة غير معتادة. الـ Invariants (account-centric، لا `profile_id`، لا User، لا Profile، لا publishing workflow) مطبَّقة بصرامة واتساق عبر كل الوثائق الـ13 — هذا مستوى انضباط نادر. الـ tech stack مكتمل وحديث، والطبقات (controller → service → repository) واضحة، وثوابت التحقق (validation constants) محسومة بأرقام فعلية.

**لكن** ادعاء ملف `implementation/README.md:5` بأن المشروع **"READY FOR IMPLEMENTATION"** و"frozen" و"passed integrity audit" هو ادعاء **مُبالَغ فيه**. يوجد بالفعل:

- **3 مشاكل مانعة (Blockers)** ستوقف العمل فعليًا عند المراحل 5–7، إحداها (دورة حذف الحساب) تمسّ صحة النظام والامتثال القانوني.
- **5 مشاكل متوسطة** تسبب تعقيدًا أو تناقضًا يجب حسمه.
- مجموعة ملاحظات بسيطة.

الخبر الجيد: **ولا واحدة من المشاكل المانعة تمنع بدء المراحل 1–4** (التهيئة، طبقة البيانات، المصادقة، الحساب/اسم المستخدم). يمكنك البدء فورًا في بناء الأساس بينما تُحَل هذه المشاكل بالتوازي قبل الوصول إلى مراحلها.

## 2. نقاط القوة (لماذا المشروع يستحق البناء)

1. **انضباط الـ Identity model.** مبدأ "Account هو الجذر الوحيد، لا User ولا Profile ولا `profile_id`" مطبَّق حرفيًا في كل ملف: الـ data model، service contracts، api contracts، validation، caching، events. لا يوجد تسرّب واحد. هذا أصعب شيء على الفِرق الحفاظ عليه، وهنا محسوم.
2. **الفصل بين الطبقات صريح.** `Controller → Service → Repository → DB`، مع منع وصول الـ controllers للـ Prisma مباشرة، ومنع الخدمات من الوصول لمستودعات نطاقات أخرى (`04-service-contracts.md`).
3. **ثوابت التحقق محسومة بقيم فعلية** (`07-validation-rules.md`): OTP = 6 أرقام/10 دقائق/5 محاولات/60 ثانية cooldown، username `^[a-z0-9]{3,30}$`، public_id `^[a-z0-9]{8}$`. لا غموض.
4. **نموذج الأمان متين** (`08-security-model.md`): trust boundary واضح، hashing للـ OTP والـ refresh tokens، rate limiting محدد لكل endpoint، منع تسريب storage keys.
5. **الخطة مرحلية مع بوابات قبول** ("Complete when…") لكل مرحلة، وبترتيب اعتمادية صحيح.
6. **الصدق التقني في وثيقة الأحداث** (`06-event-contracts.md`): تعترف صراحة بأن NestJS EventEmitter غير موثوق/in-process/بلا retry، وتوجّه للـ BullMQ عند الحاجة لضمان التنفيذ. هذا نضج هندسي.

## 3. المشاكل المانعة (Blockers) — يجب حلّها قبل مراحلها

### 🔴 B1 — تناقض Rendering ↔ Out Links: تتبّع النقرات معطّل كما هو موصّف

هذا أخطر اكتشاف.

- المعمارية المجمّدة تنص: **"Button Renderer → reads block-stored label and Out Link reference"** (`rendering/rendering.architecture.canon.v1.md:485`). أي أن الـ renderer **يحتاج مرجع الـ Out Link** ليبني الرابط.
- لكن `implementation/04-service-contracts.md:587-595` **يمنع** `RenderingService` من `OutLinkRepository`، و`05-api-contracts.md:1450` ينص أن الـ render يُجمَّع من `ProfileContent, Blocks, ConnectedAccounts` **فقط**.
- وفي الـ data model، محتوى الـ button block هو `{label, url}` (`03-canonical-data-model.md:392`) — **لا يوجد حقل يربط الـ block بالـ `public_id` الخاص بالـ OutLink**. والـ OutLink يخزّن `block_id` لكن الـ Rendering ممنوع من قراءته.

**النتيجة:** لا يوجد أي مسار يجعل زر الملف العام يشير إلى `/out/{public_id}`. الأزرار ستُرسَم بالرابط الخام المباشر، فيتجاوز نطاق Out Links بالكامل، و**حدث `link_click` لن يُسجَّل أبدًا**. أي أن نطاق Out Links + تحليلات النقرات — وهو سبب وجود النطاق أصلًا — غير وظيفي بالمواصفات الحالية.

**القرار المطلوب (اختر واحدًا):**
- (أ) السماح لـ `RenderingService` بقراءة OutLinks (read-only) + تعديل `04` و`05`، أو
- (ب) تخزين `out_link_public_id` داخل الـ block أو ضمن render object يُبنى بطبقة أعلى تجمع البيانات قبل الـ rendering.

**يؤثر على:** `03`, `04`, `05`, وملفات الـ rendering. **يجب الحسم قبل المرحلة 6/7.**

---

### 🔴 B2 — دورة حذف الحساب بلا مالك تنفيذي صالح (مشكلة صحة وامتثال)

- `AccountService.deleteAccount` مطلوب منه حذف (hard delete) لـ: Sessions، OutLinks، AnalyticsEvents، ProfileContent، Blocks، ConnectedAccounts، والأصول الثنائية، ثم `Account.status='deleted'` (`04:103-108`, `03:692-694`).
- لكن الاعتماديات الممنوعة لـ `AccountService` تشمل: `AuthRepository, BlockRepository, ConnectedAccountRepository, OutLinkRepository, AnalyticsRepository, StoragePlatform` (`04:129-138`)، والوصول لمستودعات نطاق آخر ممنوع عمومًا (`04:816`).
- آلية التنسيق الوحيدة الموصّفة هي حدث `account.deleted` → معالج إبطال الجلسات فقط (`06:238`). **لا شيء يربط حذف Blocks/OutLinks/ConnectedAccounts/Analytics/Assets** بأي معالج.
- والأحداث غير دائمة وبلا retry (`06:40-43`) — غير صالحة لضمان دورة حذف حرجة.
- إضافةً: `03:692` يصنّف حذف الحساب كعملية تتطلب **transaction واحدة**، لكنها تمتد عبر نطاقات متعددة + حذف أصول غير متزامن → **لا يمكن أن تكون atomic** كما هو موصوف.

**النتيجة:** أهم عملية من ناحية الخصوصية/القانون (حذف بيانات المستخدم بشكل كامل ونهائي) **غير قابلة للتنفيذ** تحت القواعد المعلنة. هذا ليس تفصيلًا — حذف ناقص للبيانات = مخاطرة GDPR حقيقية.

**القرار المطلوب:** تعريف **deletion orchestrator/saga** صريح (إما خدمة تنسيق مخوّلة باستثناء، أو معالجات حذف يملكها كل نطاق وتُنفَّذ عبر BullMQ بضمان)، مع توصيف ترتيب التنفيذ ومعالجة الفشل الجزئي. **يجب الحسم قبل المرحلة 4.**

---

### 🔴 B3 — لا يوجد كيان تخزين لأصول الـ `image` block

- محتوى الـ image block هو `{ image_id: string }` (`03:388`, `07:269`)، والـ Storage يعالج أصول الـ image block إلى WebP (`11:73`).
- **لكن لا يوجد أي كيان في الـ data model يخزّن بيانات/مفتاح أصل الصورة.** الكيانات العشرة لا تتضمن أي Asset/ImageAsset. الـ avatar وحده له حقل (`ProfileContent.avatar_storage_key`).
- ولا يوجد أمر `uploadImage` في `ProfileService` (يوجد `uploadAvatar` فقط، `04:471-473`)، ولا endpoint رفع صورة في الخطة (المرحلة 5 تذكر avatar فقط).

**النتيجة:** الـ `image_id` يشير إلى لا شيء. الـ **image block غير قابل للعمل**.

**القرار المطلوب:** إما إضافة كيان `ImageAsset`(account_id, storage_key, …) + أمر رفع + endpoint، أو **إخراج الـ image block من نطاق V1** صراحةً. **يجب الحسم قبل المرحلة 5.**

## 4. المشاكل المتوسطة (تعقيد/تناقض يجب حسمه)

### 🟠 M1 — ربط Social Icons بالـ Out Links غير محسوم
الـ social_icons block يحوي N حسابات (`03:389`)، لكن `createOutLink` يأخذ `destination_url` واحدًا (`04:622`)، والخطة تُنشئ out-link واحدًا لكل block (`13:188`). كما أن Social Icons Renderer يقرأ ConnectedAccounts مباشرة (canon Category 3، السطر 500) — ما يعني أن نقرات الأيقونات الاجتماعية تتجاوز التتبّع. هل تُتتبَّع نقرات السوشيال؟ إن نعم: يلزم N out-links لكل block + آلية ربط في الـ render. القرار غير موجود.

### 🟠 M2 — RenderingService يحتاج StoragePlatform لكنه ممنوع منه
الـ render يبني رابط الـ avatar عبر `StoragePlatform.buildPublicUrl` (`11:111-117`, `13:200`)، لكن `StoragePlatform` ليست ضمن اعتماديات `RenderingService` المسموحة (`04:577-585`). تناقض مباشر — الإصلاح سهل (إضافتها للمسموح) لكنه حاليًا غير متّسق.

### 🟠 M3 — مزوّد إرسال البريد/OTP غير معرّف
البريد OTP طريقة مصادقة أساسية. `requestOtp` "يرسل عبر notification service" + وظيفة `auth.otp.send` (`04:176`, `13:110`)، لكن الـ tech stack (`01`) **لا يذكر أي مزوّد بريد** (SES/SendGrid/…) ولا يوجد عقد `NotificationService`. هذا معطِّل تشغيلي لمسار المصادقة الأساسي (قابل للتوصيل لكنه غير موصّف). يجب حسمه قبل المرحلة 3.

### 🟠 M4 — نص دورة الحذف في `04` يذكر كيانات غير موجودة
`04:105` يسرد ضمن الـ cascade: `AppearanceState`, `QR records`, `AI decisions` — وكلها **ليست كيانات** (الـ appearance حقل، الـ qr_config حقل JSONB، لا كيان AI — كما يؤكد `03:694` صراحةً). بقايا تصميم قديم تسبب لبسًا. يلزم تنظيف.

### 🟠 M5 — تناقض حذف AnalyticsEvent
`03:553` ينص: "تُحذف سجلات Analytics بواسطة وظائف الاحتفاظ فقط — لا حذف يدوي"، وبلا FK كي لا يتعاقب الحذف (`03:537`). لكن دورة حذف الحساب **تحذف AnalyticsEvents** (`03:694`, `04:105`). تناقض داخلي يجب حسمه (وله بُعد خصوصية: ماذا يحدث لتحليلات حساب محذوف؟).

## 5. ملاحظات بسيطة

- **m1 — مزوّد/موديل AI غير محدد:** `11` يعرّف الواجهة فقط، الـ stack لا يذكر موديلًا. غير معطِّل (AI اختياري ويتحلّل إلى مصفوفة فارغة) لكنه ناقص.
- **m2 — مكتبة توليد QR غير محددة** (`11:74,109`). بسيطة.
- **m3 — أسماء أحداث مختصرة في الخطة/README** (`block.created`, `connected-account.created/deleted` في `13:162`) مقابل الأسماء المعتمدة (`profile.block.added`, `connected-account.added/removed` في `06`). تجميلي لكنه قد يضلّل المنفّذ — وحّد التسمية.
- **m4 — عدم اتساق نسب إبطال الكاش:** `09` ينسب الإبطال لعمليات `ProfileService` (التي لا تستطيع قراءة الـ username لتكوين `profile:public:{username}`)، بينما `06:237` ينسبه — بشكل صحيح وعملي — لمعالج أحداث في نطاق Rendering. وحّد على نموذج المعالج.
- **m5 — لا حدود/ترقيم صفحات** لاستعلامات التحليلات أو لعدد الـ content blocks (غير محدود). مقبول لـ V1 لكن انتبه لأداء الـ render مستقبلًا.

## 6. هل المشروع واضح وصريح بما يكفي؟

**نعم، بدرجة عالية.** نطاق المنتج، حدوده ("Minime Is / Is Not")، رحلة المستخدم، ملكية النطاقات، والـ non-goals — كلها صريحة ولا تترك مجالًا للتأويل. أي مهندس (أو AI) سيفهم *ماذا* يُبنى و*ماذا لا* يُبنى. الغموض المتبقي ليس في *ماذا*، بل في *كيف* تتفاعل ثلاث نقاط تكامل محددة (B1, B2, M1) — وهي ثغرات تكامل بين النطاقات، لا غموض في الرؤية.
