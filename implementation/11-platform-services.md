## Storage Platform

### Responsibilities

The Storage Platform is responsible for:

- File ingestion: MIME type validation, file size enforcement
- Format canonicalization: WebP for images, SVG for QR codes
- Asset persistence in Object Storage
- Asset lifecycle: predecessor deletion on replacement
- Public URL construction from opaque storage keys
- Orphaned asset detection and cleanup (internal, async)

The Storage Platform is not responsible for:

- Business authorization — the calling service must verify Account ownership before invoking Storage Platform
- Metadata storage — the calling service persists only the returned storage key
- User media library — no library or gallery exists
- Rendering decisions — URL construction is the only rendering-adjacent responsibility

---

### Storage Format Canon

| Asset Type | Canonical Format | Notes |
|---|---|---|
| Avatar image | WebP | Original upload (JPEG, PNG, GIF, WebP) is processed to WebP and discarded |
| Image block asset | WebP | Same processing rules as avatar; original upload discarded |
| QR code | SVG | Generated from canonical profile URL; not from user-uploaded content |

No other storage formats are supported in V1. Original uploads are temporary and must not be persisted.

---

### Asset Lifecycle Rules

- Each account owns at most one avatar asset at a time.
- Uploading a new avatar replaces the current one: ProfileService calls `StoragePlatform.delete(old_key)` before persisting the new key.
- If deletion of the old key fails, the upload may still proceed; the orphaned key will be cleaned up by the Storage Platform's internal orphan cleanup process.
- No user media library exists. There is no upload-and-reuse capability in V1.
- QR code assets are generated by the system, not uploaded by the user.

---

### Storage Key Contract

Storage keys are opaque strings returned by the Storage Platform.

- Callers must not construct, derive, or guess storage keys.
- Callers must not expose storage keys in API responses. See `08-security-model.md` — Object Storage Security.
- Callers store the returned key in the domain entity field (`ProfileContent.avatar_storage_key`).
- Callers pass the stored key back to `StoragePlatform.delete` or `StoragePlatform.buildPublicUrl` when needed.

---

### Public URL Construction

Avatar and asset URLs are constructed via `StoragePlatform.buildPublicUrl(key)`. This is the only approved method for producing client-visible asset URLs.

**Pattern:** `<STORAGE_CDN_BASE>/<key>` where `STORAGE_CDN_BASE` is an environment-configured value.

**Account deletion asset cleanup:** During the account deletion cascade, `ProfileService.deleteAccountData` calls `StoragePlatform.delete(avatar_storage_key)` if set. This executes within the `account.deletion.cascade` BullMQ job. Storage Platform orphan cleanup handles any assets that escape the explicit deletion call.

**QR code assets:** The generated SVG storage key is persisted in `Account.qr_config.storage_key`. `StoragePlatform.buildPublicUrl(storage_key)` produces the downloadable QR asset URL.

**Usage pattern for rendering:**

```
ProfileContent.avatar_storage_key
  → StoragePlatform.buildPublicUrl(key)
  → avatar_url field in render object
```

Direct key interpolation must not be used. URL construction must always go through `buildPublicUrl`.

---

### StoragePlatform Interface

```
StoragePlatform.upload(file: Buffer, mimeType: string, assetType: 'avatar' | 'image' | 'qr'):
  Promise<{ key: string }>
  — Validates MIME type against assetType allowlist.
  — Enforces maximum file size (environment-configured).
  — Converts to canonical format (WebP for images, SVG for QR codes).
  — Returns opaque storage key.
  — Rejects executable file types unconditionally.

StoragePlatform.delete(key: string): Promise<void>
  — Physically removes the asset from Object Storage.
  — Called during: account deletion cascade, avatar replacement.
  — Idempotent: deleting a non-existent key must not throw.

StoragePlatform.buildPublicUrl(key: string): string
  — Returns the CDN-prefixed public URL for the key.
  — Used only during render object assembly or download URL generation.
  — Must not be stored in the database; build fresh at response time.
```

---

### Storage Platform Prohibitions

- Product domain code must not construct Object Storage keys or URLs directly.
- Product domain code must not store raw storage keys in public API responses.
- Product domain code must not store original-format (pre-WebP) uploads.
- Product domain code must not implement a media library, gallery, or asset version history.
- Product domain code must not call Storage Platform before business authorization has been verified.

## AI Platform

### Responsibilities

The AI Platform is responsible for:

- Generating username suggestions based on user context
- Generating social account setup suggestions
- Generating basic profile improvement suggestions
- Generating onboarding guidance for new users

The AI Platform is not responsible for:

- Applying any changes to business data (user confirmation is always required)
- Determining platform correctness (Data Platform is authoritative, never AI)
- Blocking product domain operations on AI failure
- Storing accepted decisions (accepted decisions result in normal domain mutations; AI Platform has no separate storage)

---

### V1 AI Scope

V1 AI capabilities are limited to:

```
Username suggestions        — candidate usernames presented during registration
Social setup assistance     — suggested social platforms/handles during connected account setup
Profile improvement         — optional suggestions for bio, display name, or profile structure
Onboarding guidance         — step-by-step onboarding assistance for new accounts
```

The following are not AI Platform responsibilities in V1 and must not be implemented:

```
Automatic profile publication
Automatic content generation without user review
AI-driven analytics interpretation
AI-driven rendering decisions
AI-driven security decisions
```

---

### User Confirmation Rule

AI responses must never modify business state automatically.

Every AI-generated suggestion must be presented to the user for review. The user must take an explicit action (confirm, reject, or modify) before any business mutation occurs.

If the user confirms an AI suggestion, the resulting mutation executes through the normal Product Domain service (AccountService, ProfileService, etc.) with standard validation and event emission. The AI Platform does not own the mutation.

---

### AIPlatform Interface

```
AiPlatform.suggestUsernames(context: { display_name?: string, interests?: string[] }):
  Promise<string[]>
  — Returns candidate usernames; not applied automatically.
  — Failure returns empty array; never throws to caller.

AiPlatform.suggestSocialSetup(context: { platform?: string, profile_hint?: string }):
  Promise<Array<{ platform: SocialPlatform, suggested_username: string }>>
  — Returns candidate social account configurations.
  — Failure returns empty array; never throws to caller.

AiPlatform.suggestProfileImprovements(context: { display_name: string, bio?: string }):
  Promise<Array<{ field: string, suggestion: string }>>
  — Returns suggested profile content changes.
  — Failure returns empty array; never throws to caller.

AiPlatform.generateOnboardingGuidance(context: { account_id: string, completed_steps: string[] }):
  Promise<OnboardingStep[]>
  — Returns onboarding steps relevant to current account state.
  — Guidance only; does not mutate business state.
  — Failure returns empty array; never throws to caller.
```

---

### AI Platform Security Rules

Per `08-security-model.md` — AI Security:

- AI responses are treated as untrusted input. Validate before presenting to the user.
- AI responses must never be applied to business state without explicit user confirmation.
- AI responses must never appear in security-sensitive contexts (authentication, authorization decisions).
- AI Platform failures must be logged but must not interrupt product domain operations.

---

### AI Platform Prohibitions

- AI Platform must not own any database entity in V1.
- AI Platform must not be invoked in the authentication or authorization flow.
- AI Platform must not bypass validation rules defined in `07-validation-rules.md`.
- AI Platform must not apply business mutations directly; all mutations execute through domain services.
- Product domain correctness must never depend on AI Platform availability.
