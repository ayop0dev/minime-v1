## Storage Platform

### Responsibilities

The Storage Platform is responsible for:

- File ingestion: MIME type validation, file size enforcement
- Format canonicalization: WebP for images, SVG for QR codes
- Asset persistence in Object Storage
- Asset lifecycle: predecessor deletion on replacement
- Public URL construction from opaque storage keys
- Orphaned asset detection and cleanup (internal, async)

The Storage Platform is not responsible for:

- Business authorization — the calling service must verify Account ownership before invoking Storage Platform
- Metadata storage — the calling service persists only the returned storage key
- User media library — no library or gallery exists
- Rendering decisions — URL construction is the only rendering-adjacent responsibility

---

### Storage Format Canon

| Asset Type | Canonical Format | Notes |
|---|---|---|
| Avatar image | WebP | Original upload (JPEG, PNG, GIF, WebP) is processed to WebP and discarded |
| Image block asset | WebP | Same processing rules as avatar; original upload discarded |
| QR code | SVG | Generated internally from the account's `qr_url` (`https://minime.ae/qr/{qr_code_id}`); never from user-uploaded content. PNG is a derived export/download format only and is never the canonical stored asset. |

No other storage formats are supported in V1. Original uploads are temporary and must not be persisted.

---

### Asset Lifecycle Rules

- Each account owns at most one avatar asset at a time.
- Uploading a new avatar replaces the current one in this order — **upload → validate → DB swap → delete old (async)** — never the reverse: (1) `StoragePlatform.upload` stores the new file and returns `new_key`; (2) the new file is validated (per `07-validation-rules.md` — ImageAsset constants); (3) `ProfileContent.avatar_storage_key` is updated to `new_key` in a single DB write — this is the moment the new avatar becomes live; (4) only after step 3 commits does ProfileService call `StoragePlatform.delete(old_key)`. The old key is never deleted before the new key is durably referenced.
- If step 1 or 2 fails, `ProfileContent.avatar_storage_key` is untouched — the account keeps its current avatar, and the rejected upload is not persisted or referenced anywhere.
- If step 4 (deleting the old key) fails, the DB swap has already succeeded, so the account is never left without an avatar; the orphaned old key is cleaned up by the Storage Platform's internal orphan cleanup process instead.
- No user media library exists. There is no upload-and-reuse capability in V1.
- QR code assets are generated by the system, not uploaded by the user. Exactly one QR asset exists per Account at a time, referenced by `AccountQRCode.canonical_asset_id`.
- A new QR asset replaces the previous one only during system-triggered regeneration (missing/corrupted asset recovery, storage migration, format migration). User-triggered QR regeneration is forbidden.

---

### Storage Key Contract

Storage keys are opaque strings returned by the Storage Platform.

- Callers must not construct, derive, or guess storage keys.
- Callers must not expose storage keys in API responses. See `08-security-model.md` — Object Storage Security.
- Callers store the returned key in the domain entity field (`ProfileContent.avatar_storage_key`).
- Callers pass the stored key back to `StoragePlatform.delete` or `StoragePlatform.buildPublicUrl` when needed.

---

### Public URL Construction

Avatar and asset URLs are constructed via `StoragePlatform.buildPublicUrl(key)`. This is the only approved method for producing client-visible asset URLs.

**Pattern:** `<STORAGE_CDN_BASE>/<key>` where `STORAGE_CDN_BASE` is an environment-configured value.

**Account deletion asset cleanup:** During the account deletion cascade, `ProfileService.deleteAccountData` calls `StoragePlatform.delete(storage_key)` for each ImageAsset owned by the account, and calls `StoragePlatform.delete(avatar_storage_key)` if set. Both execute within the `account.deletion.cascade` BullMQ job. Storage Platform orphan cleanup handles any binary assets that escape the explicit deletion calls.

**QR code assets:** The generated SVG storage key is persisted in `AccountQRCode.canonical_asset_id` (a separate Account-owned record, not a field on Account; see `03-canonical-data-model.md` — entity 12). `StoragePlatform.buildPublicUrl(canonical_asset_id)` produces the downloadable QR asset URL.

**Usage pattern for rendering:**

```
ProfileContent.avatar_storage_key
  → StoragePlatform.buildPublicUrl(key)
  → avatar_url field in render object
```

Direct key interpolation must not be used. URL construction must always go through `buildPublicUrl`.

---

### StoragePlatform Interface

```
StoragePlatform.upload(file: Buffer, mimeType: string, assetType: 'avatar' | 'image' | 'qr'):
  Promise<{ key: string, mime_type: string, width: number, height: number, file_size: number }>
  — Validates MIME type against assetType allowlist.
  — Enforces maximum file size (environment-configured).
  — Converts to canonical format (WebP for images, SVG for QR codes).
  — Returns opaque storage key and processed asset metadata (mime_type, width, height, and file_size reflect the stored output after format conversion).
  — Rejects executable file types unconditionally.

StoragePlatform.delete(key: string): Promise<void>
  — Physically removes the asset from Object Storage.
  — Called during: account deletion cascade, avatar replacement, system-triggered QR asset regeneration.
  — Idempotent: deleting a non-existent key must not throw.

StoragePlatform.buildPublicUrl(key: string): string
  — Returns the CDN-prefixed public URL for the key.
  — Used only during render object assembly or download URL generation.
  — Must not be stored in the database; build fresh at response time.
```

---

### Storage Platform Prohibitions

- Product domain code must not construct Object Storage keys or URLs directly.
- Product domain code must not store raw storage keys in public API responses.
- Product domain code must not store original-format (pre-WebP) uploads.
- Product domain code must not implement a media library, gallery, or asset version history.
- Product domain code must not call Storage Platform before business authorization has been verified.

## QR Code System

QR Code is an implementation area within the Account domain — never a standalone Product Domain and never a Platform Service. It is documented here because, like Storage and AI, it has an explicit internal-service contract worth making concrete. Full architectural definition: `Architecture-Product-Definition/qr-code/qr-code.system.specification.v1.md`, `qr-code.generation.specification.v1.md`, `qr-code.lifecycle.specification.v1.md`.

### Responsibilities

The QR Code System is responsible for:

- Generating exactly one QR Code record and SVG asset per Account, mandatorily, before the Account becomes active — never deferred
- Persisting the QR Code record as Account-owned data (`AccountQRCode`, entity 12)
- Resolving `GET /qr/{qr_code_id}` to the Account's current public profile route by reading `username` live from the Account record, and redirecting to `/{username}`
- Providing the existing asset for download and share

The QR Code System is not responsible for:

- QR customization (color, style, logo, branding) — not a V1 capability
- QR scan analytics or tracking — not a V1 capability
- Regeneration triggered by the user — forbidden in all cases
- Routing or rendering the public profile itself — that remains owned by Routing/Rendering
- Storing `username` or any username-derived data — username is never persisted on the QR Code record

---

### QRService

`QRService` is an internal Account-domain utility, not a Platform Service and not a separate Product Domain. It is called only by `AccountService`.

```
QRService.generate(account_id: string, qr_code_id: string, qr_url: string): Promise<{ svg: Buffer }>
  — Produces the canonical SVG QR representation of qr_url using an internal QR library or equivalent implementation dependency.
  — No external provider, no provider abstraction, no external API call, no vendor lock, no per-generation cost, no rate-limit dependency.
  — Deterministic for a given qr_url.
  — Performs no persistence; AccountService stores the result via StoragePlatform.
```

### QR Generation Flow

```text
Account created
↓
Canonical public profile route created
↓
QRService.generate(account_id, qr_code_id, qr_url)
↓
SVG QR asset generated
↓
Asset saved through Storage Platform
↓
QR Code record saved under Account-owned data (AccountQRCode)
↓
Account becomes active
```

This flow runs exactly once per Account, orchestrated by `AccountService.createQrCode` (see `04-service-contracts.md`), and must complete successfully — record persisted, canonical SVG asset persisted — before the Account reaches `status = 'active'`. If any step fails, the Account must not become active. There is no lazy-generation fallback triggered by a later dashboard view, settings view, download, share, or public profile visit.

---

### QR Asset Format

- The canonical stored QR asset format is always SVG.
- PNG is supported only as an export/download format, derived on demand from the canonical SVG asset.
- PNG export/download must never create a second `AccountQRCode` record and must never become the canonical stored format.

---

### QR Route Resolution

```
AccountService.resolveQrCode({ qr_code_id }): Promise<{ username: string } | null>
  — Called only by the public GET /qr/{qr_code_id} redirect route.
  — Looks up AccountQRCode by qr_code_id; resolves account_id; reads Account.username live through account_id (the QR Code record never stores username); verifies the Account is active and resolves to an active public profile route.
  — Resolution flow: qr_code_id → AccountQRCode → account_id → Account.username → redirect to /{username}.
  — Returns null (→ 404) if the record does not exist, the Account is deleted/suspended, or no active public profile route resolves.
  — Never exposes internal Account data. Never renders a separate public profile page.
```

---

### QR Regeneration

System-triggered regeneration is allowed only for: missing asset recovery, corrupted asset recovery, storage migration, format migration. It is invoked via `AccountService.regenerateQrAsset` and must preserve `qr_code_id`, `account_id`, and `qr_url`. It must never create a new `AccountQRCode` record. There is no user-facing trigger for this operation.

---

### Mandatory Creation Before Account Activation

```text
An Account cannot reach active status unless its AccountQRCode record and canonical SVG asset have been successfully created and persisted.
```

- QR Code creation is mandatory for every active Account; it must occur during the account creation transaction or as a required blocking post-creation step before the Account becomes active.
- If `AccountQRCode` record creation fails, the Account must not become active.
- If canonical SVG asset generation or persistence fails, the Account must not become active.
- An active Account without a QR Code record must never exist.
- An active Account whose QR Code record references a missing canonical SVG asset must never exist.
- Lazy QR generation on first dashboard view, first settings view, first download, first share, or first public profile visit is forbidden.

---

### QR Code Account Settings Presentation

- The QR Code is visible only inside `Settings / Account / QR Code`, displayed directly — no separate `View` action exists.
- Supported user actions: Download, Share, Copy profile link. Nothing else.
- Download and Share read the existing `AccountQRCode` record and asset; they never generate a new record and never mutate the existing one.
- `View`, `Print`, `Customize`, `Regenerate`, `Delete`, and `Edit destination` must never be exposed as user actions.

---

### QR Code System Prohibitions

- QR Code must never become a standalone Product Domain or a Platform Service.
- QR Code must never be keyed by `username`. It is always keyed by `account_id` (`UNIQUE(account_id)`).
- `qr_code_id` must always be globally unique (`PRIMARY KEY(qr_code_id)` or `UNIQUE(qr_code_id)`). No two `AccountQRCode` records may share a `qr_code_id`, and no two Accounts may share an `AccountQRCode` record.
- The `AccountQRCode` record must never store `username` or a username snapshot. Resolution always reads `Account.username` live through `account_id`.
- The QR Code payload must never contain anything other than `https://minime.ae/qr/{qr_code_id}` — never JSON, account data, profile data, authentication data, or embedded user information.
- An external QR generation provider or provider abstraction must never be introduced.
- User-triggered regeneration, replacement, reset, customization, or deletion must never be implemented.
- QR scan analytics and QR scan tracking must never be implemented in V1.
- An Account must never reach active status without a successfully created and persisted `AccountQRCode` record and canonical SVG asset.
- Lazy QR generation must never be implemented at any trigger point after Account creation.

## AI Platform

### Responsibilities

The AI Platform is responsible for:

- Executing on-demand profile Analysis Sessions when the user explicitly requests analysis ("Analyze My Profile")
- Constructing context from canonical account state before each Analysis Session
- Delegating inference to a single configured Provider via `Provider.execute(request)`
- Returning profile improvement suggestions for user review — never applying changes automatically
- Persisting the Analysis Session as a formal V1 entity (see `03-canonical-data-model.md` — entity 11)

The AI Platform is not responsible for:

- Applying any changes to business data (user confirmation is always required)
- Username suggestions during registration (not a V1 AI capability)
- Social account setup suggestions (not a V1 AI capability)
- Onboarding guidance (not a V1 AI capability)
- Background analysis or scheduled AI tasks (no background AI in V1)
- AI chat or conversational flows (not a V1 capability)
- Determining platform correctness (Data Platform is authoritative, never AI)
- Blocking product domain operations on AI failure
- Storing accepted decisions, suggestion history, or learned preferences (no AI persistence in V1)

---

### V1 AI Scope

V1 AI capabilities are limited to a single feature:

```
Analyze My Profile    — on-demand Analysis Session; triggered only by explicit user request
```

An Analysis Session:
- Is triggered by an explicit "Analyze My Profile" user action
- Builds the Analysis Input Snapshot from current account state (profile, blocks, connected accounts, analytics summary)
- Computes an Input Hash from the normalized snapshot
- Reuses the latest completed Analysis Session for the account when its Input Hash, `analysis_version`, and `output_schema_version` all match the current configured values (Analysis Session Reuse) — never applies, modifies, or re-executes when reused
- Otherwise creates a new Analysis Session, builds an `AIProviderRequest`, and calls `Provider.execute(request)`
- Returns profile improvement suggestions for user review
- Does not execute or apply any change to business data

Analysis validity depends only on the Input Hash, `analysis_version`, and `output_schema_version` all matching. Time elapsed since the last Analysis Session never determines whether a new analysis is required. A matching Input Hash with a differing `analysis_version` or `output_schema_version` never qualifies for reuse — a new Analysis Session is created.

The following are explicitly excluded from V1 and must not be implemented:

```
Username suggestions             — not a V1 AI capability
Social setup suggestions         — not a V1 AI capability
Onboarding guidance              — not a V1 AI capability
Background AI analysis           — not a V1 capability
AI chat or conversational flows  — not a V1 capability
Automatic profile publication    — not a V1 capability
Automatic content generation     — not a V1 capability
AI-driven analytics interpretation  — not a V1 capability
AI-driven rendering decisions    — not a V1 capability
AI-driven security decisions     — not a V1 capability
```

### Analysis Session Entity

The Analysis Session is an official V1 platform entity — a persisted record of one profile analysis execution, not merely a response payload. Full field-level definition: `03-canonical-data-model.md` — entity 11.

Conceptual structure:

```
AnalysisSession = {
  id
  account_id
  input_hash
  analysis_version          // semantic version string, e.g. "1.0.0"
  output_schema_version     // semantic version string, e.g. "1.0.0"
  status
  provider_key              // diagnostic metadata only
  model_key                 // diagnostic metadata only
  report
  scores
  suggestions
  recommendations
  metadata
  created_at
  completed_at
}
```

Field-level Prisma definition (types, indexes, constraints): `03-canonical-data-model.md` — entity 11.

Rules:

- Belongs to exactly one Account.
- Created only after an explicit "Analyze My Profile" user request.
- Represents one profile analysis execution.
- Stores one completed AI analysis result.
- Multiple Analysis Sessions may exist for one Account.
- Only `status = 'completed'` Analysis Sessions may be reused. Failed Analysis Sessions must never be reused.
- **Final Analysis Session Reuse lookup:** find the latest `AnalysisSession` where `account_id` = current account, `status = 'completed'`, `input_hash` = current Input Hash, `analysis_version` = current configured analysis version (exact string match), and `output_schema_version` = current configured output schema version (exact string match). Only that session may be reused. Any mismatch creates a new Analysis Session.
- `analysis_version` (`string`, semantic version `MAJOR.MINOR.PATCH`, e.g. `"1.0.0"`) identifies the version of Minime's own analysis business logic — prompt meaning, scoring logic, recommendation logic, structured output interpretation, and analysis input rules. It is owned by Minime, not the Provider. It is not a model version and not a provider version. Changing the configured Provider or Model does not automatically change `analysis_version`; changing the analysis's business meaning must change it.
- `output_schema_version` (`string`, semantic version `MAJOR.MINOR.PATCH`, e.g. `"1.0.0"`) identifies the structured output schema Minime uses to validate and store the AI response. It is owned by Minime, not the Provider. It is not a model version and not a provider version. Changing the output JSON structure must change it.
- **Recommended (not enforced as governance) version interpretation:** patch (`x.y.PATCH`) for minor prompt wording, typo fixes, or non-breaking output clarification; minor (`x.MINOR.y`) for non-breaking additions or scoring/recommendation refinements; major (`MAJOR.y.z`) for breaking changes in analysis meaning, scoring interpretation, or output structure. This is interpretive guidance, not a release-management process.
- `AIService` must use the currently configured `analysis_version` and `output_schema_version` whenever checking for a reusable Analysis Session.
- `provider_key` and `model_key` are diagnostic metadata only, useful for debugging, support, monitoring, cost analysis, provider performance review, and incident investigation. They must never participate in Analysis Session Reuse, analysis validity, AI output validation, user-facing logic, Product Domain decisions, comparison of analysis meaning, or deciding whether a new analysis is required.
- A change to the configured Provider or Model never invalidates an existing Analysis Session by itself. If a Provider or Model change must invalidate prior sessions, that must be expressed explicitly by incrementing `analysis_version` or `output_schema_version` (whichever reflects the reason) — never inferred from `provider_key` or `model_key`.
- Analysis Session history is not AI learning. Analysis Sessions are stored reports only — not accepted suggestions, not rejected suggestions, not learned preferences. Those remain V2 scope.

---

### Analysis Input Snapshot

The Analysis Input Snapshot is the canonical, deterministic set of account-owned data used to generate the Input Hash and the `AIProviderRequest`. It is the only approved V1 source of AI analysis input.

```
AnalysisInputSnapshot = {
  account: {
    id: string,
    username: string,
    status: string
  },
  profile_content: {
    display_name: string | null,
    bio: string | null,
    contact_information: Record<string, unknown> | null
  },
  connected_accounts: Array<{
    platform: string,
    username: string,
    url: string,
    sort_order: number
  }>,
  blocks: Array<{
    type: string,
    sort_order: number,
    content: Record<string, unknown>,
    settings: Record<string, unknown>
  }>,
  appearance: Record<string, unknown> | null,
  analytics_summary: {
    total_views: number,
    total_clicks: number,
    ctr: number,
    device_breakdown: Record<string, number>,
    top_links: Array<{ label: string, click_count: number }>
  } | null
}
```

`analytics_summary` may be `null` when analytics are unavailable at execution time. This does not introduce any analytics requirement beyond existing V1 analytics scope (`04-service-contracts.md` — AnalyticsService).

**Exclusions.** The snapshot must never include: transient UI state, runtime cache data, Sessions, audit logs, raw Provider output, previous Analysis Sessions, accepted or rejected AI suggestions, or learned preferences.

**Construction and hashing rules:**

- The snapshot must be generated exclusively by `AIService`. Product Domains must not provide custom ad-hoc AI context.
- The snapshot must be normalized before hashing: array elements ordered deterministically (e.g. `connected_accounts` and `blocks` by `sort_order`; `top_links` by a fixed, stable ordering), object keys serialized in a deterministic order, and empty values (`null`, `[]`, `{}`) given a stable representation.
- `input_hash` is computed from the normalized snapshot. Any change to snapshot content — and only a change to snapshot content — must produce a different `input_hash`. The specific hash algorithm is an implementation detail; the snapshot is a cache-equivalence key, not a security boundary.
- Previous Analysis Sessions must never be part of the snapshot.
- Accepted/rejected suggestion history and AI learning remain V2 scope and are excluded from the snapshot.

---

### User Confirmation Rule

AI responses must never modify business state automatically.

Every AI-generated suggestion must be presented to the user for review. The user must take an explicit action (confirm, reject, or modify) before any business mutation occurs.

If the user confirms an AI suggestion, the resulting mutation executes through the normal Product Domain service (ProfileService, etc.) with standard validation and event emission. The AI Platform does not own the mutation.

### AIService Interface

`AIService` is the single application entry point for AI. Product domain services must not invoke `Provider.execute` directly.

```
AIService.analyzeProfile(account_id: string): Promise<
  | { status: 'completed', analysis_session_id: string, reused: boolean, report, scores, suggestions, recommendations, completed_at }
  | { status: 'failed', analysis_session_id: string, suggestions: [] }
  | { status: 'disabled', suggestions: [] }
>
  — Triggered only by explicit "Analyze My Profile" user request.
  — If AI_API_KEY is absent or empty: returns { status: 'disabled', suggestions: [] } immediately; no Analysis Session row is created at all — this is distinct from 'failed', which always has a persisted session. See implementation/05-api-contracts.md — Part 9 for why this distinction matters to the caller.
  — Builds the AnalysisInputSnapshot for account_id; normalizes it; computes its Input Hash.
  — Finds the latest Analysis Session for account_id where status = 'completed', input_hash matches, analysis_version matches the configured current analysis_version, and output_schema_version matches the configured current output_schema_version.
  — On full match (Analysis Session Reuse): returns { status: 'completed', reused: true, ... } using that Analysis Session's stored result; does not call Provider.execute.
  — On no match (input_hash, analysis_version, or output_schema_version differs, or no prior session exists): creates a new Analysis Session (status: 'pending', analysis_version: current, output_schema_version: current), builds an AIProviderRequest, and calls Provider.execute(request) once.
  — On success: validates the structured output against the configured current output_schema_version; persists the Analysis Session as status: 'completed' with report/scores/suggestions/recommendations populated; returns { status: 'completed', reused: false, ... }.
  — On any provider failure, timeout, or output that does not conform to the current output_schema_version: persists the Analysis Session as status: 'failed'; returns { status: 'failed', analysis_session_id, suggestions: [] }; never throws to caller.
  — AIService never selects, names, or switches models or providers; it only builds the snapshot, prepares input, calls Provider.execute(request), validates structured output, and persists the Analysis Session.

AIService.getLatestCompletedAnalysisSession(account_id: string): Promise<AnalysisSession | null>
  — read-only; never calls Provider.execute and never creates or mutates a session
  — returns the most recent status = 'completed' Analysis Session for account_id by completed_at, or null if none exists
  — backs GET /api/v1/account/ai/analysis/latest (implementation/05-api-contracts.md — Part 9)
```

### AIProviderRequest

`AIProviderRequest` is the request contract passed to `Provider.execute`. It is intentionally minimal — only what V1 needs.

```
AIProviderRequest = {
  system_prompt: string,
  user_prompt: string,
  response_format: string,
  max_output_tokens: number,
  temperature: number,
  metadata: {
    analysis_version: string,
    output_schema_version: string,
    account_id: string
  }
}
```

`AIService` builds the `AIProviderRequest` from the Analysis Input Snapshot and the prompt templates in `packages/config/ai-prompts.ts`. `metadata.analysis_version` and `metadata.output_schema_version` are the currently configured Minime-owned semantic version strings; `metadata.account_id` identifies the owning account. `AIService` never includes provider names or model names anywhere in the request, including `metadata` — that selection belongs entirely to the configured Provider.

---

### Provider.execute Pattern

`Provider.execute` is the single execution boundary between `AIService` and the configured AI provider.

```
Provider.execute(request: AIProviderRequest): Promise<string>
  — Executes one inference request against the configured provider and model.
  — The Provider alone selects the configured provider implementation and the configured model. AIService never selects, names, switches, or reads model configuration.
  — Provider is selected by AI_PROVIDER; model is selected by AI_MODEL_ID. Both are read only inside the Provider implementation.
  — Replacing the provider requires only env var change and a compatible adapter. No domain service code changes.
  — Applies AI_TIMEOUT_MS timeout; rejects on timeout (AIService converts to a failed Analysis Session and empty suggestions).
  — One Provider and one Model are active at runtime. Multi-provider routing is not a V1 capability.
```

---

### AI Platform Security Rules

Per `08-security-model.md` — AI Security:

- AI responses are treated as untrusted input. Validate before presenting to the user.
- AI responses must never be applied to business state without explicit user confirmation.
- AI responses must never appear in security-sensitive contexts (authentication, authorization decisions).
- AI Platform failures must be logged but must not interrupt product domain operations.

---

### AI Platform Provider Configuration

V1 AI provider is configured entirely through environment variables. No provider configuration is stored in the database.

**Required environment variables:**

| Variable | Description | Default |
|---|---|---|
| `AI_PROVIDER` | Provider identifier. Fully implementation-configurable. No specific vendor is required. | see `.env.example` |
| `AI_MODEL_ID` | Model ID within the configured provider. Fully implementation-configurable. | see `.env.example` |
| `AI_API_KEY` | Provider API key. Never logged or exposed. If absent: AI is disabled; `AIService` returns empty suggestions. | — (required) |
| `AI_TIMEOUT_MS` | Maximum milliseconds to wait for a `Provider.execute` response. | `10000` |

**Prompt templates** live in `packages/config/ai-prompts.ts`. They are application code, not database entities. They must not be user-editable.

**`analysis_version` and `output_schema_version`** are Minime-owned application constants, not environment variables. They are semantic version strings (`MAJOR.MINOR.PATCH`, e.g. `"1.0.0"`), co-located with the prompt templates in `packages/config/ai-prompts.ts`, and incremented manually by engineers when analysis business meaning or the output schema shape changes. They are never provider-configured or model-configured, and changing `AI_PROVIDER` or `AI_MODEL_ID` never changes them.

**`provider_key` and `model_key`** are diagnostic metadata only — captured from the Provider at execution time (e.g. the active provider/model identifiers) so engineers can debug, support, monitor, analyze cost, review provider performance, and investigate incidents. `AIService` may receive and persist these diagnostic values as returned by the Provider, but never reads, interprets, or acts on them for any business purpose.

**Operational rules:**
- If `AI_API_KEY` is absent or empty: AI is disabled. `AIService.analyzeProfile` returns `{ suggestions: [] }`; no error is thrown.
- All `Provider.execute` calls must time out after `AI_TIMEOUT_MS`. Timeout marks the Analysis Session as failed and returns empty suggestions; never throws.
- Provider and model are selected entirely via `AI_PROVIDER` and `AI_MODEL_ID`, read only inside the Provider implementation. No specific vendor is architecturally required. Replacing the provider requires only env var changes and a compatible adapter implementation; no domain service code changes.
- `analysis_version` and `output_schema_version` must be valid semantic version strings (`MAJOR.MINOR.PATCH`). Recommended (not enforced as governance) interpretation: patch for minor prompt wording/typo/non-breaking output clarification changes; minor for non-breaking additions or scoring/recommendation refinements; major for breaking changes in analysis meaning, scoring interpretation, or output structure.
- Input Hash is computed from the normalized Analysis Input Snapshot. The latest completed Analysis Session whose `input_hash`, `analysis_version`, and `output_schema_version` all match the current configured values (exact string match) is reused (Analysis Session Reuse) without executing a new `Provider.execute` call. Reuse depends only on this three-way match — never on elapsed time, and never on `provider_key` or `model_key`. A matching `input_hash` with a differing `analysis_version` or `output_schema_version` does not qualify for reuse.
- AI output is validated against the configured current `output_schema_version`. Output that does not conform marks the Analysis Session as failed.
- A Provider or Model change (`AI_PROVIDER`/`AI_MODEL_ID`) never invalidates existing Analysis Sessions by itself. If a Provider or Model change must invalidate prior sessions, that intent must be expressed explicitly by incrementing `analysis_version` or `output_schema_version`.
- AI Platform failures must be logged but must not interrupt product domain operations.

### AI Platform Prohibitions

- AI Platform must not own any database entity in V1.
- AI Platform must not be invoked in the authentication or authorization flow.
- AI Platform must not bypass validation rules defined in `07-validation-rules.md`.
- AI Platform must not apply business mutations directly; all mutations execute through domain services.
- Product domain correctness must never depend on AI Platform availability.
- `AIService` must not be invoked from background jobs, event handlers, or scheduled tasks. AI is strictly on-demand in V1.
- Multi-provider routing must not be implemented in V1.
- AI Platform must not persist suggestion history, decision signals, or learned preferences. No `AiDecision`, `AcceptedDecision`, or `SuggestionHistory` entity exists in V1. `AnalysisSession` is the only AI-owned entity.
- `AIService` must never select models, know model names, know provider names, switch models, or read model configuration. The configured Provider exclusively owns provider-specific logic, the configured provider implementation, configured model selection, and provider communication.
- Product Domains must never call `Provider.execute` directly. Product Domains call `AIService` only.
- Analysis validity must never be determined by elapsed time. Only a matching Input Hash, `analysis_version`, and `output_schema_version` together determine whether an existing Analysis Session is reused.
- `analysis_version` and `output_schema_version` must never be owned, set, or interpreted by the AI Provider. They must never be derived from `AI_PROVIDER` or `AI_MODEL_ID`.
- `AIProviderRequest.metadata` must never contain a provider name or model name.
- An Analysis Session whose `output_schema_version` does not match the current configured value must never be reused, even if `input_hash` and `analysis_version` match.
- `provider_key` and `model_key` must never participate in Analysis Session Reuse, analysis validity, AI output validation, user-facing logic, Product Domain decisions, comparison of analysis meaning, or the decision of whether a new analysis is required. They are diagnostic metadata only (debugging, support, monitoring, cost analysis, provider performance review, incident investigation).
- A change to `provider_key` or `model_key` (i.e. a Provider or Model change) must never by itself invalidate, expire, or trigger re-creation of an existing Analysis Session. Invalidation due to changed meaning must be expressed only through `analysis_version` or `output_schema_version`.
- This refinement must not introduce an AI gateway, AI router, provider factory, model router, dynamic model selection, agent framework, tool router, additional background workers, or additional abstraction layers.
