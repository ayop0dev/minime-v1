## Purpose

Defines the canonical backend service contracts for Minime V1.

Service contracts define:

- service ownership
- commands and queries
- allowed and forbidden dependencies
- emitted events

This document does not define HTTP endpoints.

This document does not define database schema.

---

## Global Service Rules

### Implementation

- Services must be NestJS injectable providers.
- Services must contain business use case coordination only.
- Services must access persistence only through owned repositories.
- Services may call Platform Services when required.
- Services may publish internal events after successful business operations.
- Services must not access Prisma directly.
- Controllers must not access Prisma directly.
- Repositories must return typed results.
- Services must not contain HTTP request or response logic.
- Services must not depend on controllers.
- Services must not own another Product Domain's business decisions.

### Commands

- Commands must validate input before mutation.
- Commands must persist changes through repositories.
- Commands must return the latest saved state.
- Commands must publish events only after successful persistence.

### Queries

- Queries must not mutate business state.
- Queries must apply ownership filters.
- Queries must exclude soft-deleted records by default.

### Cross-Service Dependencies

- Product Domain services may depend on another Product Domain service only when this document explicitly allows it.
- Platform Services may be used by any Product Domain service.
- Platform Services must not depend on Product Domain services.

### Service Lifetime

- Services must be stateless.
- Request-specific state must not be stored inside services.

## AccountService

**Owner:** Account domain

**Purpose:** Own account lifecycle, settings, and the QR Code entry point. QR Code is an implementation area within Account, never a standalone Product Domain.

### Commands

```
createAccount({account_id, username})
  — creates Account + ProfileContent only, as one step inside a database transaction opened and owned by the caller; AccountService never creates, reads, or touches AuthenticationIdentity (see Forbidden Dependencies) — see "Registration Transaction Coordination" under AuthService for the full registration sequence and ownership boundary
  — account_id is generated by the caller (AuthService.completeRegistrationWithOAuth) and passed in; AccountService never generates the account identifier for a registration
  — ProfileContent.display_name is initialized to username (the same value being reserved), never to any provider-reported name; this is the only source rule for the initial value — see profile.content.specification.v1.md
  — called only by AuthService.completeRegistrationWithOAuth, never called standalone and never exposed as a direct user-facing command
  — does not call createQrCode itself; the coordinator (completeRegistrationWithOAuth) calls createQrCode({account_id}) after its own transaction commits, per "Registration Transaction Coordination"

updateAccountSettings({account_id, settings})
  — mutates Account.settings
  — if `gtm_container_id` changes: invalidates `profile:public:{username}` (see `09-caching-strategy.md` — Cache Invalidation) since it is rendered into the cached public HTML response

createQrCode({account_id})
  — internal command; not a direct user-facing HTTP endpoint; never user-triggered
  — called exactly once per account, immediately after createAccount establishes the canonical public profile route, and must complete successfully before the Account becomes active
  — generates qr_code_id, builds qr_url = https://minime.ae/qr/{qr_code_id}
  — calls QRService.generate(account_id, qr_code_id, qr_url) to produce the canonical SVG asset
  — stores the asset via StoragePlatform.upload(assetType: 'qr')
  — persists one AccountQRCode record {qr_code_id, account_id, qr_url, destination_type: 'public_profile', canonical_asset_id, canonical_format: 'svg'} — no username or username snapshot is stored
  — must never be called a second time for the same account_id (enforced by UNIQUE(account_id) and PRIMARY KEY(qr_code_id))
  — on failure (record creation or asset generation/persistence), the Account must not become active

regenerateQrAsset({account_id, reason})
  — internal command; system-triggered only — never user-triggered; not exposed via any user-facing endpoint
  — valid reasons: missing_asset | corrupted_asset | storage_migration | format_migration
  — regenerates the SVG asset via QRService.generate using the existing qr_code_id and qr_url; stores it via StoragePlatform.upload
  — updates AccountQRCode.canonical_asset_id and updated_at only; qr_code_id, account_id, and qr_url never change

deleteQrCode({account_id})
  — internal command; called only by the account.deletion.cascade job
  — if an AccountQRCode record exists: calls StoragePlatform.delete(canonical_asset_id), then hard-deletes the AccountQRCode record
  — idempotent: safe to re-run

deleteAccount({account_id})
  — immediate and final; there is no grace period and no cancellation in V1
  — AccountService.deleteAccount executes, in one database transaction: (1) Account.status = 'deleted'; (2) INSERT AccountDeletionOutbox({account_id, requested_at: now()}) — see 03-canonical-data-model.md — AccountDeletionOutbox
  — after the transaction commits: calls AuditLogService.record({account_id, action: 'account_deleted', entity_type: 'account', entity_id: account_id}), then publishes account.deleted via EventEmitter as a best-effort notification (cache invalidation and any other non-critical observer — see 06-event-contracts.md)
  — the durable guarantee that cross-domain cascade cleanup happens does not come from the EventEmitter publish in the previous step; it comes from the AccountDeletionOutbox row committed in step (1)-(2), which the Deletion Outbox Dispatcher job (10-background-jobs.md — Job 8) guarantees will result in `account.deletion.cascade` being enqueued even if the process crashes immediately after commit and the EventEmitter publish never happens
  — AccountService must not directly invoke repositories outside its allowed dependencies; cross-domain cleanup is delegated to the account.deletion.cascade BullMQ job (Job 7), whose enqueue is guaranteed by Job 8, not by EventEmitter
  — account record is retained permanently (account_id referenced by AuditLog rows for the audit retention period, and by the AccountDeletionOutbox row permanently)
  — username permanently reserved; not released in V1
  — cascade order, idempotency, and partial failure rules: see 10-background-jobs.md — Job 7 and Job 8
```

There is no `updateQrCodeConfiguration`, no `customizeQrCode`, and no user-facing `regenerateQrCode`. The QR Code is never user-editable.

### Queries

```
getAccountById({id})
getAccountByUsername({username})
getAccountSettings({account_id})
getQrCode({account_id})
  — returns the account's AccountQRCode record and its asset URL (via StoragePlatform.buildPublicUrl); read-only, never generates or mutates
resolveQrCode({qr_code_id})
  — called only by the public GET /qr/{qr_code_id} redirect route
  — looks up AccountQRCode by qr_code_id; resolves account_id; reads Account.username live through account_id (never from a value stored on the QR Code record); verifies the Account is active and has a resolvable public profile route
  — returns {username} for redirect, or a not-found result if the record does not exist, the Account is deleted/suspended, or no active public profile route resolves
  — never returns internal Account data
```

```
StoragePlatform            — QR asset upload/delete/buildPublicUrl only
AuditLogService            — write-only, after createAccount and deleteAccount succeed

```

```

```

### Emitted Events

```
account.created
account.settings.updated
account.qr.created
account.deleted
```

There is no `account.qr.updated` event. The QR Code record is created once and never mutated by the user; system-triggered asset regeneration does not emit a business event.

### Implementation Rules

- AccountService must own account lifecycle.
- AccountService must own settings mutations.
- AccountService must own the QR Code entry point as an implementation area, never as a standalone Product Domain.
- `createAccount` never opens its own transaction; it executes as one step inside the single database transaction opened by `AuthService.completeRegistrationWithOAuth` (Account + ProfileContent + AuthenticationIdentity + UsernameReservation deletion — see "Registration Transaction Coordination" under AuthService). `createQrCode` runs as a required blocking step immediately after that transaction commits, once the canonical public profile route exists — never deferred, never lazy.
- An Account must never reach `status = 'active'` unless its `AccountQRCode` record and canonical SVG asset have been successfully created and persisted. If `createQrCode` fails for any reason, the Account must not become active.
- Lazy QR creation triggered by first dashboard view, first settings view, first download, first share, or first public profile visit is forbidden.
- Every Account has exactly one `AccountQRCode` record, enforced by `UNIQUE(account_id)`. Every `AccountQRCode` record belongs to exactly one Account, enforced by `PRIMARY KEY(qr_code_id)`. `createQrCode` must never run more than once per account.
- The QR Code record is keyed by `account_id`, never by `username`. The record never stores `username` or a username snapshot. The QR redirect flow always resolves `qr_code_id → AccountQRCode → account_id → Account.username → /{username}` live.
- `qr_code_id` and `qr_url` are permanent and immutable once created.
- AccountService must not mutate ProfileContent fields beyond creating the initial record.
- `deactivate` and `reactivate` operations do not exist in V1. `suspended` status is administrative and is not user-initiated.
- There is no `cancelAccountDeletion`. Deletion is immediate and final per architecture.
- `deleteAccount` sets Account.status = 'deleted' and inserts an `AccountDeletionOutbox` row in the same transaction; only after that transaction commits does it write the `AuditLog` row and publish `account.deleted` (best-effort). AccountService must not call repositories of other domains; cross-domain cascade executes via the `account.deletion.cascade` BullMQ job (Job 7), whose enqueue is durably guaranteed by the Deletion Outbox Dispatcher (Job 8) reading `AccountDeletionOutbox`, never by the EventEmitter publish alone.
- Username is permanently reserved after deletion; it is not released in V1.

## AuthService

**Owner:** Authentication domain

**Purpose:** Own authentication flows, session lifecycle, and registration completion.

### Provider Adapter Contract

`AuthService` dispatches all provider-specific work to a resolved `AuthenticationProviderAdapter`. Core `AuthService` commands never branch on `provider` for any reason other than adapter resolution — every Google-specific or Apple-specific behavior (token endpoint URLs, `client_secret` construction, claim shapes) lives inside the adapter, never inside `AuthService` itself.

```
type AuthenticationProvider = "google" | "apple"   // V1 allowlist; extended only by adding an adapter + allowlist entry in a future version

type AuthenticationProviderAdapter = {
  provider: AuthenticationProvider

  buildAuthorizationUrl(input: { state: string, nonce: string, redirect_uri: string, intent: "authenticate" | "link" }):
    { authorization_url: string }

  exchangeCodeForTokens(input: { code: string, redirect_uri: string }):
    { id_token: string, access_token?: string, refresh_token?: string }   // access/refresh token are never persisted; see 08-security-model.md

  validateIdToken(input: { id_token: string, nonce: string }):
    { provider_subject: string, email?: string, email_verified?: boolean, raw_claims: Record<string, unknown> }
    // signature, issuer, audience, expiry, and nonce are all validated inside this step

  normalizeIdentity(input: { validated_identity: ReturnType<validateIdToken>, token_result: ReturnType<exchangeCodeForTokens> }):
    { provider, provider_subject, email?, email_verified?, provider_profile?: Record<string, unknown> }   // NormalizedAuthenticationIdentity — the only shape the rest of AuthService ever reads
}

resolveProviderAdapter(provider: AuthenticationProvider): AuthenticationProviderAdapter
  — internal function; rejects (before any provider call) if provider is not in the V1 allowlist
  — returns the GoogleAuthenticationProviderAdapter or AppleAuthenticationProviderAdapter; both implement the same contract above
  — adding a future adapter (e.g. Instagram, Facebook, Microsoft, GitHub, LinkedIn, X, TikTok) means implementing this contract and adding the value to the allowlist — it requires no change to AuthService's public commands, no change to API routes, and no change to the AuthenticationIdentity shape
```

Apple's `response_mode=form_post` callback delivery is normalized inside the Apple adapter / the request-normalization step before `handleOAuthCallback` is invoked — see "Callback Transport Normalization" in `authentication.policy.v1.md`. `handleOAuthCallback` itself always receives the same `{provider, code, state}` shape regardless of provider.

### Commands

```
startOAuth({provider, intent, account_id?})
  — provider: must be in the V1 allowlist ("google" | "apple"); unknown values are rejected before resolveProviderAdapter is ever called
  — intent: "authenticate" (default, unauthenticated) | "link" (requires an authenticated account_id)
  — resolves the AuthenticationProviderAdapter for provider
  — generates a single-use, unguessable state value and a nonce; persists them server-side with a short TTL, bound to provider, intent, and (for "link") account_id — this stored binding is the "state context" referenced throughout this document
  — calls adapter.buildAuthorizationUrl({state, nonce, redirect_uri, intent})
  — returns {authorization_url, state}

handleOAuthCallback({provider, code, state})
  — looks up the persisted state context by state; rejects on missing/expired/already-consumed state (CSRF + replay protection)
  — validates that provider (from request data) matches the provider bound to the state context; rejects on mismatch — this is enforced before resolveProviderAdapter is called
  — rejects if provider is not in the V1 allowlist (defense in depth; startOAuth already enforces this, but callback re-validates independently)
  — resolves the AuthenticationProviderAdapter for provider
  — calls adapter.exchangeCodeForTokens({code, redirect_uri}) — backend-to-provider call only; token exchange never happens client-side
  — calls adapter.validateIdToken({id_token, nonce}) — signature, issuer, audience, expiry, and nonce are all checked inside this call; a failure here rejects the request and extracts no identity data
  — calls adapter.normalizeIdentity(...) to produce a NormalizedAuthenticationIdentity {provider, provider_subject, email?, email_verified?, provider_profile?}
  — performs duplicate-identity detection: looks up AuthenticationIdentity by (provider, provider_subject) — see "Identity Merge Policy" in authentication.policy.v1.md; provider_subject is the only identity key, email is never consulted for this lookup
  — intent "authenticate": if a matching identity exists, creates a Session and returns {account_id}; if none exists, returns {needs_registration: true, oauth_handoff_token} where oauth_handoff_token is a short-lived, single-use, signed token carrying the NormalizedAuthenticationIdentity — never the raw provider tokens
  — intent "authenticate" + matching identity already belongs to the account bound to a prior session in this same browser context: not applicable here (re-authentication metadata refresh is intent-independent; see refreshOAuthIdentityMetadata below)
  — intent "link": requires the authenticated account_id bound to the state context at startOAuth time; if (provider, provider_subject) is already bound to a different account, rejects with a duplicate-identity error; if it already belongs to the same account, calls refreshOAuthIdentityMetadata instead; otherwise returns {link_handoff_token} — a short-lived, single-use, signed token carrying {account_id, NormalizedAuthenticationIdentity}, consumed by linkAuthenticationProvider
  — never persists provider access tokens or refresh tokens

completeRegistrationWithOAuth({reservation_id, oauth_handoff_token})
  — provider and the NormalizedAuthenticationIdentity fields arrive only via a previously validated oauth_handoff_token, never as raw client input
  — validates oauth_handoff_token: signature, expiry, single-use (reject if already consumed)
  — validates reservation exists and has not expired
  — is the sole transaction coordinator for registration; see "Registration Transaction Coordination" below for the exact sequence, ownership, and failure handling
  — creates Session and issues tokens only after the coordination sequence below completes successfully (i.e. after `createQrCode` succeeds)

### Registration Transaction Coordination

`completeRegistrationWithOAuth` is the single coordinator for registration. No other command creates an Account, a ProfileContent row, or an AuthenticationIdentity row as part of registration. The sequence is:

1. Generate `account_id` (coordinator-owned; neither AccountService nor AuthRepository generates it).
2. Open one database transaction. Within it, in order:
   a. Acquire the same Postgres advisory transaction lock keyed by `hashtext(normalized_username)` that `UsernameService.reserveUsername` acquires (see UsernameService.reserveUsername for why this specific lock, and not just the two tables' own UNIQUE constraints, is required). Re-validate that `username_from_reservation` still exists as a non-expired `UsernameReservation` while holding the lock — if it has expired or been consumed concurrently, abort before step (b).
   b. Call `AccountService.createAccount({account_id, username: username_from_reservation})` — creates Account + ProfileContent (see AccountService.createAccount).
   c. Create the AuthenticationIdentity row directly via AuthRepository: `{account_id, provider, provider_subject, email, email_verified, provider_profile}`. AuthService is the only owner of AuthenticationIdentity creation; AccountService never creates it.
   d. Delete the UsernameReservation via UsernameReservationRepository.
3. Commit the transaction. If any step in 2 fails, the transaction rolls back as a whole and no Account, ProfileContent, AuthenticationIdentity, or reservation-deletion survives; the reservation remains valid until its normal expiry and the client may retry.
4. After commit, call `AccountService.createQrCode({account_id})` as a required blocking step (this is I/O against QRService/StoragePlatform and cannot participate in the step-3 database transaction).
5. If `createQrCode` fails: execute a compensating deletion in a new transaction — hard-delete the AuthenticationIdentity, ProfileContent, and Account rows created in step 2 for this `account_id` (in that order, respecting FKs). The UsernameReservation is not restored; the client must submit a new reservation to retry. The endpoint returns a registration failure; no Session is created and no tokens are issued.
6. If `createQrCode` succeeds, the Account is active. Only then does `completeRegistrationWithOAuth` call `AuditLogService.record({account_id, action: 'account_created', entity_type: 'account', entity_id: account_id})`, publish `account.created` (best-effort), create a Session, and issue tokens.

linkAuthenticationProvider({account_id, link_handoff_token})
  — link_handoff_token is the token returned by handleOAuthCallback(intent: "link"); its bound account_id must match the authenticated caller's account_id — reject on mismatch
  — validates link_handoff_token: signature, expiry, single-use (reject if already consumed)
  — creates a new AuthenticationIdentity row for the account using the token's NormalizedAuthenticationIdentity payload
  — rejects if (provider, provider_subject) is already bound to a different account (duplicate-identity detection was already performed in handleOAuthCallback; this is a defense-in-depth re-check)
  — does not affect any other linked identity
  — after the row is created, calls AuditLogService.record({account_id, action: 'authentication_method_added', entity_type: 'authentication_identity', entity_id: auth_identity_id})

unlinkAuthenticationProvider({account_id, auth_identity_id})
  — verifies the AuthenticationIdentity row identified by auth_identity_id belongs to account_id; rejects on ownership mismatch
  — rejects if this is the account's last remaining AuthenticationIdentity (an account must always have at least one)
  — deletes the AuthenticationIdentity row
  — does not affect the account, its data, or any other linked identity
  — after the row is deleted, calls AuditLogService.record({account_id, action: 'authentication_method_removed', entity_type: 'authentication_identity', entity_id: auth_identity_id})

refreshOAuthIdentityMetadata({account_id, provider, normalized_identity})
  — invoked when handleOAuthCallback resolves to an identity that already belongs to the calling/authenticated account (i.e. the user re-authenticated with a provider they already linked)
  — updates only email, email_verified, and provider_profile on the existing AuthenticationIdentity row, from normalized_identity
  — provider_subject is never changed by this operation

createSession({account_id})
  — creates Session record (stores refresh_token_hash)
  — issues short-lived access token (JWT, not stored)
  — issues refresh token (rotated on every use)

refreshSession({refresh_token})
  — validates refresh token against stored hash
  — rejects revoked or expired sessions
  — additionally rejects if the session's Account.status is not 'active' (via the same cached check described in 08-security-model.md — "Per-Request Account Status Check") — this is necessary because Session.revoked_at may not be set yet during the bounded window between account deletion and the deletion cascade's session-cleanup step (10-background-jobs.md — Job 7 step 1); relying on Session.revoked_at alone would let a deleted account mint a new access token during that window
  — rotates refresh token (old hash invalidated, new hash stored)
  — issues new access token

revokeSession({session_id, account_id})
  — sets Session.revoked_at

revokeAllSessions({account_id})
  — sets revoked_at on all active, non-expired sessions for the account
```

There is no `requestOtp`, `verifyOtp`, `completeRegistrationWithOtp`, or any Email OTP registration/login method. There is no generic, open-ended `provider` value — `provider` is constrained to `google` and `apple` only at every call site, validated against the V1 allowlist before any adapter is resolved.

### Implementation Rules

- Authentication methods, supported providers, and token security: See `08-security-model.md` and `07-validation-rules.md`.
- AuthService must not own profile content.
- AuthService must not implement Email OTP, direct email authentication, password authentication, or AuthEmailDelivery.
- AuthService must not store provider access tokens or refresh tokens.
- `provider` must be constrained to `google` and `apple` at every command boundary in V1. AuthService must reject any other value before resolving a Provider Adapter or calling a provider.
- AuthService must contain no Google-specific or Apple-specific branching beyond `resolveProviderAdapter`. All provider-specific logic (token endpoints, claim shapes, `client_secret` construction) lives inside the adapter for that provider.
- A future provider is added by implementing a new `AuthenticationProviderAdapter` and extending the allowlist. This must never require changes to `AuthService`'s public command signatures, to API route structure, or to the `AuthenticationIdentity` data model shape.
- Every account must have at least one `AuthenticationIdentity` at all times. `unlinkAuthenticationProvider` must reject removal of the last remaining identity.
- `unlinkAuthenticationProvider` identifies the identity to remove by `auth_identity_id`, never by `provider` alone, and must verify the identity belongs to the calling account before deleting it.
- `provider_subject` is the only authentication identity key. `email` must never be used to find, merge, or match an existing account — see "Identity Merge Policy" in `authentication.policy.v1.md`. There is no automatic account merge and no manual account merge in V1.
- Linking and unlinking never imply Social Account ownership verification and never interact with the Connected Accounts domain.
- A `primary_provider` display preference, if implemented, is a presentation-only concern; it must never be consulted for ownership, authentication, or authorization decisions, and changing it must never affect authentication validity for any linked identity.

## UsernameService

**Owner:** Username domain

**Purpose:** Own username availability checking and reservation during registration.

### Commands

```
reserveUsername({username})
  — validates format (a-z/0-9, length 3–30, lowercase only)
  — acquires a Postgres advisory transaction lock keyed by `hashtext(normalized_username)` before checking availability — the same lock, keyed identically, that `AuthService.completeRegistrationWithOAuth` acquires before creating the Account (see "Registration Transaction Coordination" — AuthService). This is what actually prevents the race `UsernameReservation.UNIQUE(username)` and `Account.UNIQUE(username)` cannot prevent on their own, since they are two independent constraints on two separate tables with no cross-table exclusion: without the shared lock, a reservation could be created for a username in the instant between another request's availability check and its registration-completing INSERT into `Account`.
  — checks availability against Account.username and active UsernameReservation (both checked while holding the lock)
  — creates UsernameReservation record, then releases the lock (transaction commit)

expireReservation({id})
  — hard deletes expired UsernameReservation
  — called by background job only; no HTTP endpoint
```

### Queries

```
checkAvailability({username})
  — validates format
  — checks Account.username and active UsernameReservation
  — returns {username, available: boolean}
```

### Allowed Dependencies

```
UsernameReservationRepository
AccountRepository              — check if username already exists on Account
EventBus
```

### Forbidden Dependencies

```
AuthRepository
ProfileContentRepository
BlockRepository
ConnectedAccountRepository
AnalyticsRepository
StoragePlatform
AIService
```

### Emitted Events

```
username.reserved
username.reservation.expired
```

### Implementation Rules

- Username is set on Account.username during registration via AuthService. It is never changed in V1.
- There is no `changeUsername` command. Username changes are not supported in V1.
- There is no `assignUsername` as a standalone operation. Assignment happens in AuthService.completeRegistration.
- There is no `releaseUsername` as a user-facing operation. Reservations expire automatically via background job.
- Availability check must consult both Account.username (permanent) and UsernameReservation (temporary hold).

## ConnectedAccountsService

**Owner:** Connected Accounts domain

**Purpose:** Own persistent storage of user-entered social links (ConnectedAccount records). Store, manage, and reorder.

### Commands

```
addConnectedAccount({account_id, platform, username})
  — calls SocialAccountsService.validateAndNormalize({platform, username})
  — persists ConnectedAccount{account_id, platform, username, url, sort_order}
  — after persistence: calls AuditLogService.record({account_id, action: 'connected_account_added', entity_type: 'connected_account', entity_id: <new id>})

updateConnectedAccount({id, account_id, username})
  — only username is editable; platform is immutable
  — calls SocialAccountsService.validateAndNormalize to regenerate url
  — opens one database transaction (see ProfileService — "Block + OutLink Transaction Coordination" for the identical pattern applied here): persists updated ConnectedAccount{username, url}; finds all social_icons blocks owned by account_id whose content.accounts array references this connected_account_id; for each affected block, archives every active OutLink for (block_id, connected_account_id) then creates the replacement OutLink with the new destination_url — all within the same transaction
  — if any step fails, the transaction rolls back as a whole: the ConnectedAccount keeps its old username/url and every OutLink touched keeps its pre-update state — never a ConnectedAccount with a new url pointing at a stale or missing OutLink
  — after the transaction commits: invalidates profile cache (profile:public:{username}); calls AuditLogService.record({account_id, action: 'connected_account_updated', entity_type: 'connected_account', entity_id: id})

removeConnectedAccount({id, account_id})
  — ownership verified via account_id before deletion
  — opens one database transaction: finds all social_icons blocks owned by account_id whose content.accounts array references this connected_account_id; removes the connected_account_id entry from each affected block's content.accounts array (BlockRepository write); archives any active OutLinks for (block_id, connected_account_id) for each affected block; then Hard Deletes the ConnectedAccount — no soft delete, no archive, no trash — all within the same transaction
  — if any step fails, the transaction rolls back as a whole: the ConnectedAccount, its Block references, and its OutLinks all remain exactly as they were before the call — never a partial state where the Block reference is removed but the ConnectedAccount or its OutLink still exists, or vice versa
  — after the transaction commits: invalidates profile cache; calls AuditLogService.record({account_id, action: 'connected_account_removed', entity_type: 'connected_account', entity_id: id})

reorderConnectedAccounts({account_id, ordered_ids})
  — validates `ordered_ids` as a set before touching any row: no duplicate IDs; every ID belongs to account_id; the set is complete — it must contain exactly the account's current ConnectedAccount IDs, no fewer and no more (same rule as ProfileService.reorderBlocks; see 03-canonical-data-model.md — Block — "Concurrency" for the identical rationale)
  — acquires the per-account advisory lock (the same lock as Block mutations — see 03-canonical-data-model.md — ConnectedAccount — "Concurrency"), then updates sort_order for each ConnectedAccount (assigned as 0..N-1 in array order) in one transaction
  — if validation fails, no `sort_order` value is written for any row
  — after the transaction commits: calls AuditLogService.record({account_id, action: 'connected_account_reordered', entity_type: 'connected_account', entity_id: account_id})

deleteAccountConnectedAccounts({account_id})
  — internal operation; called only by account.deletion.cascade job, not by controllers or HTTP requests
  — hard-deletes all ConnectedAccount records for the account
  — idempotent: safe to re-run
```

## ProfileService

**Owner:** Profile domain

**Purpose:** Own profile content (display name, bio, avatar, contact, appearance) and block lifecycle.

### Commands

```
updateProfileContent({account_id, display_name?, bio?, contact?})
  — updates ProfileContent fields; validates fields per `07-validation-rules.md`

uploadAvatar({account_id, file})
  — stores file via StoragePlatform
  — updates ProfileContent.avatar_storage_key with returned storage key

addBlock({account_id, type, content, settings, sort_order, style_overrides?})
  — validates type against approved BlockType enum
  — validates content and settings against per-type schema
  — enforces max 1 per account for identity blocks (avatar, name, bio)
  — enforces MAX_BLOCKS_PER_ACCOUNT total active blocks per account; rejects with business error if limit is reached
  — creates Block record; style_overrides stores only explicit user-authored style keys; absent or null means full inheritance from the active Theme at render time
  — if block type is `button` or `social_icons`: the Block insert and the OutLinksService.createOutLink call(s) below execute in one database transaction opened by ProfileService (see "Block + OutLink Transaction Coordination" below); ProfileService still never touches OutLinkRepository directly — OutLinksService's repository writes simply participate in the transaction ProfileService opened
  — if block type is `button`: calls OutLinksService.createOutLink once (connected_account_id = null)
  — if block type is `social_icons`: calls OutLinksService.createOutLink once per entry in content.accounts (connected_account_id = that entry's connected_account_id)
  — if any OutLinksService.createOutLink call fails (including the partial-unique-index rejection on `(block_id, connected_account_id)` — see 03-canonical-data-model.md — OutLink), the transaction rolls back and no Block is created

updateBlock({id, account_id, content?, settings?, style_overrides?})
  — validates ownership via account_id
  — validates updated content/settings against per-type schema
  — for `button` and `social_icons` blocks, the Block mutation and any OutLink archive/create calls below execute in one database transaction opened by ProfileService — see "Block + OutLink Transaction Coordination" below
  — mutates Block.content and/or Block.settings and/or Block.style_overrides
  — style_overrides stores only explicit user-authored style keys; null clears all overrides (full inheritance from active Theme); inherited values must not be stored
  — for `button` blocks: if label or url changes, archives old OutLink then creates new OutLink, in that order, within the transaction
  — for `social_icons` blocks: compares old and new content.accounts; archives OutLinks for removed entries; creates OutLinks for added entries; entries with unchanged connected_account_id retain their OutLink; all archive/create calls for this update share the one transaction
  — if any step fails, the transaction rolls back as a whole: the Block row and every OutLink touched by this call revert to their pre-update state — there is no partial outcome where the Block reflects a new label/url but the OutLink still points to the old destination, or vice versa

deleteBlock({id, account_id})
  — validates ownership via account_id
  — in one database transaction: soft deletes Block (sets deleted_at), then archives all active OutLinks for the block (one for button; one per icon for social_icons) — see "Block + OutLink Transaction Coordination" below
  — if the OutLink archive step fails, the transaction rolls back and the Block is not deleted (never leaves a soft-deleted Block with a still-active OutLink)

### Block + OutLink Transaction Coordination

`addBlock`, `updateBlock`, and `deleteBlock` are each a transaction coordinator, symmetric to `AuthService.completeRegistrationWithOAuth` for registration (see AuthService — "Registration Transaction Coordination"): ProfileService opens one database transaction spanning `BlockRepository` (its own) and, for `button`/`social_icons` blocks, `OutLinkRepository` (via `OutLinksService`'s commands, called synchronously within that transaction). ProfileService never gains direct access to `OutLinkRepository` — the Forbidden Dependency remains in force — it only ever calls `OutLinksService`'s existing commands, which now execute inside the caller's transaction instead of opening their own. Any failure anywhere in the sequence rolls back both the Block change and every OutLink change together; there is no intermediate committed state and no compensation step is needed, because nothing is committed until the whole sequence succeeds. The database-level partial unique index `(block_id, connected_account_id) WHERE status = 'active'` (03-canonical-data-model.md — OutLink) is the defense-in-depth backstop against a concurrent second request racing the same transaction window.

reorderBlocks({account_id, ordered_ids})
  — validates `ordered_ids` as a set before touching any row: no duplicate IDs; every ID belongs to account_id and is an active (non-soft-deleted) Block; the set is complete — it must contain exactly the account's current active Block IDs, no fewer and no more (a partial list, a list with a stale/deleted ID, or a list missing a currently-active block is rejected outright, not partially applied)
  — acquires the per-account advisory lock (see 03-canonical-data-model.md — Block — "Concurrency"), then updates Block.sort_order for all listed blocks (assigned as 0..N-1 in array order) in one transaction
  — if validation fails, no `sort_order` value is written for any block

updateAppearance({account_id, selected_theme_id?, customizations?})
  — updates ProfileContent.appearance_config
  — validates selected_theme_id against the theme catalog at `packages/config/themes.ts`; rejects unknown theme IDs

uploadImage({account_id, file})
  — validates file MIME type and file size per 07-validation-rules.md — ImageAsset constants
  — uploads file via StoragePlatform (assetType: 'image'); receives back storage_key and processed dimensions
  — creates ImageAsset record {account_id, storage_key, mime_type, width, height, file_size}
  — returns {image_id, width, height, file_size, mime_type} for use in image block content
  — storage key is never returned to callers beyond this service

deleteImage({id, account_id})
  — validates ownership via account_id
  — validates no active (non-soft-deleted) Block for this account has content->>'image_id' = id
  — if referenced by an active block: rejects with business error (user must delete the block first)
  — hard-deletes ImageAsset record
  — calls StoragePlatform.delete(storage_key)

deleteAccountData({account_id})
  — internal operation; called only by account.deletion.cascade job, not by controllers or HTTP requests
  — hard-deletes all Block records for the account
  — hard-deletes all ImageAsset records for the account; calls StoragePlatform.delete(storage_key) for each
  — hard-deletes ProfileContent record for the account (includes appearance_config JSONB field)
  — calls StoragePlatform.delete(avatar_storage_key) if avatar_storage_key is set (binary avatar asset cleanup)
  — idempotent: safe to re-run
```

```
getProfileContent({account_id})    — returns ProfileContent record
getBlocks({account_id})            — returns active Blocks ordered by sort_order
getBlock({id, account_id})         — returns one Block with ownership check
```

### Implementation Rules

- ProfileService must not implement `publishProfile` or `unpublishProfile`. There is no publishing workflow in V1.
- ProfileService does not have a `createProfile` command. ProfileContent is created by AccountService during registration.
- Saving any profile change makes it publicly visible within the cache TTL (≤ 60 seconds). There is no separate "publish" step.
- All blocks reference account_id. There is no profile_id.
- Identity blocks (avatar, name, bio) are limited to max 1 per account. This limit must be enforced on addBlock.
- Content blocks (image, social_icons, button, divider, title, textbox) are unlimited.
- avatar_storage_key stores the Storage Platform key only. Binary data lives in Object Storage.
- ImageAsset storage keys must never be returned in API responses. Public image URLs are constructed at render time via StoragePlatform.buildPublicUrl.
- A ProfileContent update that changes `display_name`, `avatar_storage_key`, `bio`, or `contact` calls `AuditLogService.record` with `action` = `display_name_updated` \| `avatar_updated` \| `bio_updated` \| `contact_information_updated` respectively, after the write succeeds. A single update touching multiple fields writes one AuditLog row per changed field.
- An image block's content.image_id must reference an ImageAsset owned by the same account. This is validated on addBlock and updateBlock.
- `uploadImage` must be called before creating or updating an image block with a new image. The returned image_id is then used in the block content.
- `deleteImage` rejects if any active block references the image. The user must delete the block first.

### Implementation Rules

- RenderingService must be stateless.
- RenderingService must never write to any repository.
- RenderingService reads directly from repositories for performance. It does not go through other domain services.
- After `getRenderedProfile` returns successfully and the public profile response is delivered, the public profile rendering pipeline must determine `device_category` using the platform device classification policy before emitting `profile.viewed`. The storage key must not appear in the rendered payload. If the referenced ImageAsset does not exist, the block must be omitted or rendered as a placeholder.
- Cache TTL and cache keys: See `09-caching-strategy.md`.
- There is no publication state. Rendering always reflects the current persisted state of profile content.

## OutLinksService

**Owner:** Out Links domain

**Purpose:** Own outbound link records, lifecycle, and click tracking initiation.

**Creation model:** OutLinks are created internally by the system. They are NOT created via user-facing API. An OutLink is created whenever a new Clickable State is created — i.e., when a Button Block or Social Icons Block is added, or when a change to label, URL, or platform creates a new Clickable State. ProfileService calls OutLinksService to create and archive OutLinks as part of block lifecycle operations.

### Commands

```
createOutLink({account_id, block_id, block_type, destination_url, label_snapshot, source_snapshot, connected_account_id})
  — internal operation; called by ProfileService, not by controllers
  — connected_account_id: null for button blocks; ConnectedAccount ID for social_icons blocks
  — triggered when a new Clickable State is created (new button block, or new social icon entry)
  — generates unique URL-safe public_id per `07-validation-rules.md` — Out Links constants
  — creates OutLink with status 'active'

archiveOutLink({id, account_id})
  — internal operation; called by ProfileService or ConnectedAccountsService, not by controllers
  — triggered when a Clickable State changes or is removed (label/url/platform change, social icon removed, block deleted)
  — validates ownership via account_id
  — sets OutLink.status = 'archived'
  — sets OutLink.archived_at = NOW()

resolveOutLink({public_id})
  — called by the public redirect route GET /out/{public_id}
  — looks up OutLink by public_id; if not found or status = 'archived', returns a not-found result (see error mapping in 05-api-contracts.md)
  — generates click_id = uuid() synchronously, before returning — this is the one and only idempotency key for this click; it is never regenerated on retry
  — enqueues the `out-links.click` BullMQ job with `jobId = click_id` and payload `{click_id, out_link_id: id, account_id}` (account_id read from the resolved OutLink row — see AnalyticsEvent.account_id snapshot on link_click, 03-canonical-data-model.md), non-blocking
  — the click is considered "counted" at the moment this enqueue call returns successfully, not at HTTP response delivery (the HTTP redirect response and the job execution are independent; a client that disconnects before receiving the 302 does not un-count a click that was already enqueued)
  — returns destination_url for redirect

purgeArchivedOutLinks()
  — background job only
  — hard deletes records where status = 'archived' AND archived_at < NOW() - 90 days

getAccountOutLinkIds({account_id})
  — internal read-only query; called only by account.deletion.cascade job before OutLink deletion
  — returns all OutLink IDs (any status) owned by account_id
  — used to collect out_link_ids before deletion so analytics cleanup does not depend on delete return values
  — idempotent: safe to call multiple times; returns empty list if OutLinks already deleted

deleteAccountOutLinks({account_id})
  — internal operation; called only by account.deletion.cascade job, not by controllers or HTTP requests
  — hard-deletes all OutLink records for the account regardless of status
  — idempotent: safe to re-run
```

### Queries

```
getOutLinks({account_id})           — returns OutLinks with status != 'archived' for account
getOutLink({id, account_id})        — returns one OutLink with ownership check
```

### Allowed Dependencies

```

```

### Forbidden Dependencies

```

```

### Implementation Rules

- OutLink has no `updateOutLink` command. All content fields (destination_url, label_snapshot, source_snapshot, block_id, block_type, public_id) are immutable after creation.
- OutLink has no `enableOutLink` or `disableOutLink`. There is no is_enabled field.
- OutLink has no `deleteOutLink`. Lifecycle is: active → archived → hard purge after 90 days.
- OutLinks are not created via user-facing API. They are created internally by the system (via ProfileService calling OutLinksService) when a new Clickable State is created.
- Archiving is not user-initiated. It is triggered internally when a Clickable State changes or a block is deleted.
- Click tracking must not block redirect. `resolveOutLink` returns destination_url immediately; analytics recording is async via BullMQ.
- public_id format and constraints: See `07-validation-rules.md` — Out Links constants. Must be globally unique.

### Callers

`AuditLogService.record` is called directly (as a plain service-to-service call, not an event) by:

```
AuthService            — authentication_method_added (linkAuthenticationProvider), authentication_method_removed (unlinkAuthenticationProvider), primary_identity_changed (see authentication.policy.v1.md — Identity Merge Policy)
```

### Implementation Rules

- AuditLogService must never be called from inside an EventEmitter handler — only from the originating command handler, so that the audit record and the business change it describes are attributable to the same request.
- Retention (12 months, then hard delete) is enforced by a background job — see `10-background-jobs.md`.

## Platform Service Usage

| Platform Service | Allowed Usage |
|---|---|

**NestJS EventEmitter characteristics:**

- In-process delivery (same Node.js instance only).
- No guaranteed delivery across restarts.
- No automatic retry on handler failure.
- No distributed or cross-process delivery.
- For durable async work requiring retry, use BullMQ jobs.

| AuditLogRepository | Platform (write access from Account, Authentication, Connected Accounts, Profile — see AuditLogService) |

Services must access only repositories owned by their domain.

Shared reads must execute through approved service contracts.

## Service Prohibitions

Implementation must not:

- place business logic inside controllers
- access Prisma outside repositories
- expose repositories to controllers
- allow repositories to publish events
- allow repositories to call Platform Services
- allow Platform Services to own Product Domain decisions
- allow services to mutate another Product Domain through its repository
- introduce a User, Profile, PublicProfile, or SocialAccount entity or repository
- introduce a NotificationService, notification inbox, notification preferences, push notification system, or any product notification system — Minime V1 has no email delivery capability; authentication is provider-based
- reference profile_id anywhere
- implement publishProfile, unpublishProfile, or any profile publication command
- implement changeUsername (username is immutable in V1)
- implement Email OTP, phone OTP, SMS authentication, or AuthEmailDelivery
- implement Social/Connected Accounts OAuth provider connections (`connectProvider`, `refreshProviderConnection`, `revokeProviderConnection`) — distinct from Authentication's `AuthService.startOAuth`/`handleOAuthCallback`/`linkAuthenticationProvider`/`unlinkAuthenticationProvider`, which do exist; Social/Connected Accounts must never become an authentication provider
- implement social account verification (verifySocialAccount)
- claim distributed event delivery through NestJS EventEmitter
